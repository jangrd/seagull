# Why ignore `Woverride-init` and `-Woverride-init-side-effects` warnings?

A conscious decision has been made to ignore the `-Woverride-init` and
`-Woverride-init-side-effects` warnings generated by some compilers.
List of affected compilers:
    - GCC
    - clang
    - TODO: check for more

## Recommendation for handling this
Handle this by explicitly ignoring the warning in subsections of code instead
of project wide. Use `SGL_UI_START` and `SGL_UI_END` macros to wrap any UI
creation code.

```c
SGL_UI_START

SGL_Element* = ELEMENT(...);

SGL_UI_END
```

If you really want to you can also just compile with `-Wno-override-init` and
`-Wno-override-init-side-effects` flags, though i still recommend only using
them on the translation unit that does the UI creation.


## What triggers `-Woverride-init`

The warning is thrown when C's compound literal is overriden by the translation
unit at compile time. Example:

```c
typedef Struct {
    int a;
    int b;
} foo;

foo my_foo = {
    .a = 10,
    .b = 20,
    .a = 30 // this overrides previous value at compile time
}
```

## Why seagull uses this behavior

Along with macros, this trick allows the library to have default values for the
creation of SGL_Element at compile time. To keep the same functionality without
triggering the warning the library would have to:
    a) compromise on the user-facing syntax
or
    b) handle these things in runtime

As far as I know the trick is defined behavior for most if not (almost) all compilers
